1. PURPOSE
----------------------------------------------------------------
This experiment supports the Ribault paper's revision after the ICFP
reviewers (#116A, #116B, #116C) flagged: (i) the evaluation as narrow
(only mergesort and text-search benchmarks), (ii) missing comparison
against modern tagged-dataflow systems (Dryad / Naiad / Timely Dataflow
/ Materialize), (iii) suspicion that TALM's reported speedup came from
the FFI-to-C leaf rather than from the dataflow scheduler (#116C: "swap
the parallelism for a spark or strategies-based implementation,
presumably it would perform quite similarly").

We answer (i) by adding an LCS wavefront benchmark — a workload whose
dataflow shape (anti-diagonal dependencies, fixed grain per block) is
qualitatively different from mergesort (tree-shaped fork/join) and
text-search (embarrassingly parallel). We answer (ii) by including
Timely Dataflow as a modern descendant of the same paradigm. We answer
(iii) by holding the per-block compute primitive identical across all
Haskell variants and isolating only the inter-block scheduler.

4. WORKLOAD: LCS WAVEFRONT
----------------------------------------------------------------
Computes the Longest Common Subsequence (LCS) length between two
sequences A, B of length N each, with alphabet size 4 and characters
generated by a deterministic LCG seeded with 42 (same generator across
all variants). The LCS DP recurrence is

    L[i][j] = L[i-1][j-1] + 1                if A[i-1] == B[j-1]
              max(L[i-1][j], L[i][j-1])      otherwise

We do NOT materialise the full N x N table. Instead, the table is
tiled into a rectangular block grid of DIM_ROWS x DIM_COLS blocks
where DIM_ROWS = P and DIM_COLS = ceil(N / TARGET_BLOCK_WIDTH) (with
floor at P so the wavefront has enough breadth). With TARGET_BLOCK_WIDTH
fixed at 1000, the per-block column width stays ~constant across N,
so cache footprint per block is bounded and the total block count
grows linearly with N. Each block needs its top row (from the block
above) and its left column (from the block to its left) before it
can fire; this gives the anti-diagonal wavefront pattern.

Only boundary rows/columns of each block are kept globally, in two
shared arrays:
    hBound : (DIM_ROWS + 1) * (N + 1) elements  — horizontal boundary
    vBound : (DIM_COLS + 1) * (N + 1) elements  — vertical boundary
Memory per run: ~(DIM_ROWS + DIM_COLS) * N * 8 bytes. For N = 500000
and DIM_COLS = 500, that is ~4 GiB peak — well within the 251 GiB
host.

Each block runs a serial DP loop with two scratch rows (prev/cur of
local-column-width size), reading the incoming top/left boundaries
and writing the outgoing bottom/right boundaries. Identical algorithm
across all variants; only the language and the inter-block scheduler
differ.


5. VARIANTS AND WHY EACH WAS CHOSEN
----------------------------------------------------------------
A. seq (Haskell sequential, P=1)
   -- The denominator for the Haskell-tier variants (Ribault-Hs,
      GHC Strat). Uses a 2-row sequential DP (no blocking) backed by
      Data.Vector.Unboxed.Mutable — the same mutable-vector idiom as
      the parallel Haskell variants. This is critical: if the
      baseline used Foreign.Ptr (faster) and the parallel variants
      used Vector (slower), reported speedups would conflate the
      parallel scheduler with a compute-backend swap.
   -- variant column in CSV: "seq"

A2. seq_c (C sequential, P=1)
   -- The denominator for the C-tier variant (Ribault-C). Monolithic
      2-row DP in pure C (gcc -O3 -march=native), structurally
      identical to (A): same LCG (seed=42, alpha=4), same square N×N
      DP loop, two int rows swapped per iteration, no bounds checks
      (C has none anyway), no blocking, single thread pinned to
      core 0 via taskset. Cross-validated: produces RESULT=65385 at
      N=100k, 130837 at N=200k, 196253 at N=300k, 261674 at N=400k,
      327182 at N=500k — identical to the seq_rust and Haskell seq
      results at the corresponding N. This is the right denominator
      for Ribault-C because comparing a C numerator against a
      Haskell denominator would inflate the speedup by the GHC↔C
      compute gap, not by the dataflow scheduler.
   -- source: scripts/lcs_wavefront/lcs_seq.c
   -- build: gcc -O3 -march=native -o lcs_seq_c lcs_seq.c
   -- variant column in CSV: "seq_c"

A3. seq_rust (Rust sequential, P=1)
   -- The denominator for the Rust-tier variants (Ribault-Rust,
      Timely, Timely-Rebalanced, Sucuri). Monolithic 2-row DP in
      Rust, built with the same release profile as the parallel
      Rust crates: opt-level=3, lto="thin", codegen-units=1, no
      debug info. CRITICAL FAIRNESS NOTE: the parallel Rust leaf
      compute (Ribault-Rust, Timely) uses raw pointers + `unsafe`
      with NO bounds checking on the inner DP loop; if seq_rust
      kept Rust's default bounds checking, it would be artificially
      slow versus the parallel numerator and would inflate the
      Rust-tier speedups. seq_rust therefore mirrors the leaf
      compute exactly: `unsafe` block, raw `*const i32 / *mut i32`
      pointers, `add(..)` arithmetic, no bounds check on the
      O(N^2) inner loop. Same LCG, same square DP, same taskset
      pin. Cross-validates RESULT byte-for-byte against seq_c at
      every N.
   -- source: scripts/lcs_wavefront/lcs_seq.rs (+ Cargo.toml)
   -- build: cargo build --release  (release profile above)
   -- variant column in CSV: "seq_rust"

B. TALM-Hs (Ribault, natural Haskell super)
   -- Primary thesis variant. The user's Haskell source is compiled
      by Ribault to a TALM dataflow graph; each block is a "super"
      instruction whose body is a Haskell function. No Foreign.Ptr,
      no Foreign.Marshal.* in the user-visible super — only
      Data.Vector.Unboxed.Mutable, as a Haskell programmer would
      naturally write. Run by the Trebuchet C runtime (per-thread
      super dispatch via supers_hs_init_thread stubs; the FFI cost
      lives in the runtime boundary, not in user code).
   -- variant column in CSV: "super"

C. TALM-C (Ribault with C super body)
   -- Same Trebuchet runtime and same dataflow graph as TALM-Hs, but
      the super body is a pure C function (no Haskell, no FFI in user
      code). Isolates the language gap from the scheduler: if
      TALM-Hs ≈ TALM-C, the Haskell user code is not the bottleneck.
   -- variant column in CSV: "ribault_c"

C2. TALM-Rust (Ribault with Rust super body)
   -- Same Trebuchet runtime and same dataflow graph as TALM-Hs and
      TALM-C, but the super body is a Rust function (Rust crate built
      as `crate-type = "staticlib"`, exporting `extern "C"` s10..s13
      that override the WEAK_FN slots in supers_wrappers.c). No
      Haskell, no Python, no monad-par. Provides the language-tier
      pair for the Rust dataflow competitors: TALM-Rust vs Timely
      (both Rust per-block compute, only the scheduler differs).
      Same algorithmic safety argument as TALM-C: each block writes
      to disjoint regions of g_hBound/g_vBound, so the data race
      across worker threads is avoided algorithmically and not by
      Rust's borrow checker.
   -- variant column in CSV: "ribault_rust"

D. GHC Strategies (parList rseq)
   -- Standard Haskell parallel idiom: same block grid, anti-diagonal
      barrier via `using` parList rseq sparks. Same per-block compute
      backend as TALM-Hs (Data.Vector.Unboxed.Mutable). Tests
      the hypothesis from reviewer #116C: "swap the parallelism for a
      spark... presumably it would perform quite similarly".
   -- variant column in CSV: "strategies"

E. Sucuri + Rust (Tiago's dataflow lib)
   -- Independent dataflow runtime, included by request of the
      advisor. Python orchestrator (CPython 3.14.4t free-threaded,
      no GIL) drives a dataflow graph; the per-block compute is a
      Rust function exposed via PyO3. Tests whether a different
      research dataflow runtime — non-Trebuchet, non-Haskell — sees
      the same scaling shape on LCS.
   -- variant column in CSV: "sucuri"

F0. Timely Dataflow (Rust) — REBALANCED variant
   Identical to F (below) in every aspect except the Exchange
   function: instead of `exchange(|(i,j)| (i*DIM_COLS + j))`, the
   rebalanced version uses `exchange(|(i,j)| i)`, distributing
   blocks by their grid row. Inside any anti-diagonal d the active
   blocks are (i, d-i) for i in [lo,hi]; with the original hash the
   worker assignment becomes (i*(DIM_COLS-1) + d) mod P, which
   collapses to fewer than P distinct workers whenever
   gcd(DIM_COLS-1, P) > 1 (the diagonal serializes onto a single
   worker in the worst case, e.g. N=100k P=3 with DIM_COLS=1000:
   gcd(999,3)=3 sends every diagonal to one worker). The
   rebalanced hash distributes (i, d-i) across P workers as i mod P,
   uniform inside every diagonal regardless of gcd. The two Timely
   results are reported side by side in master.csv (variants
   `timely` and `timely_rebalanced`) so the paper can show both:
   (a) the natural / idiomatic Timely choice and the load-imbalance
   trap that comes with it, and (b) a hand-tuned exchange that
   removes the trap. The Ribault firing-rule scheduler has no
   exchange-by-hash and is structurally immune to either case.

F. Timely Dataflow (Rust)
   -- Modern descendant of Naiad / Dryad / Materialize (all three
      cited as missing from related work by reviewer #116A). Dryad
      and Naiad are research artifacts retired by Microsoft (no
      longer runnable); Timely Dataflow is the live Rust port by
      Frank McSherry (the Naiad author), used in production by
      Materialize. We use the `timely` crate v0.13 with N workers
      and an epoch-per-antidiagonal driver: worker 0 emits block
      coordinates at epoch d = i+j, an Exchange operator distributes
      them across workers, the map operator runs compute_block, and
      probe.less_than(input.time()) barriers between epochs.
      Compute backend: shared mutable boundary arrays via raw
      pointers (Rust unsafe), wrapped in Send+Sync newtype.
      Algorithmically race-free because blocks on the same
      anti-diagonal write to disjoint regions.
   -- variant column in CSV: "timely"

ALPHABET:  4

in the parallel section only (input generation and final score read
excluded — same start/stop instrumentation in every variant).


7. RUN ORDER  (N-major)
----------------------------------------------------------------
The orchestrator runs **all variants for the same N** before moving
on to the next N. This way, every time an N completes we can build
a cross-variant table for that N (saved to
$OUTROOT/N_<N>/N_<N>_master.csv) and inspect partial results without
waiting for the full experiment to finish.

For each N in {50k, 100k, 200k, 300k, 400k, 500k}:
    Phase 1: SEQ + TALM-Hs + STRAT  (run_validated.sh)
    Phase 2: TALM-C                 (run_ribault_c.sh)
    Phase 3: TALM-Rust              (run_ribault_rust.sh)
    Phase 4: Sucuri + Rust          (run_sucuri.sh)
    Phase 5: Timely Dataflow        (run_timely.sh)


8. METHODOLOGICAL EQUIVALENCE
----------------------------------------------------------------
Reviewers correctly worried that uneven compute backends would mask
the scheduler comparison. The following invariants hold across every
(variant, N, P) measurement in master.csv:

a) Same input. All variants regenerate the same A, B sequences from
   the LCG with seed=42. The expected LCS score is cross-validated
   between variants at every N (every variant emits "RESULT=<n>" and
   the runner aborts if any variant disagrees with the first).

b) Same block grid. DIM_ROWS = P, DIM_COLS = ceil(N/1000) with floor
   P, for every parallel variant. Sequential baseline is unblocked
   (single pass, two-row DP); all parallel variants use the same
   block grid for the same (N, P).

c) Same per-block compute idiom within each language tier:
     Haskell  (seq, TALM-Hs, STRAT):             Data.Vector.Unboxed.Mutable
     C        (TALM-C):                          raw int32 arrays via malloc/calloc
     Rust     (TALM-Rust, Sucuri-leaves, Timely):raw i32/i64 slices via libc::malloc
   The cross-language gap (Haskell vs C vs Rust) is intentional and
   part of the comparison; the within-language equivalence is what
   isolates the scheduler. Note the deliberate pairings:
     - Haskell tier: TALM-Hs vs STRAT (scheduler isolated)
     - Rust tier:    TALM-Rust vs Timely (scheduler isolated)
     - C tier:       TALM-C stands alone as the upper-bound for what
                     the Trebuchet scheduler can extract with the fastest
                     compute backend; no equivalent C-native dataflow
                     runtime exists.
     - Hybrid:       Sucuri (Python orchestrator + Rust leaves) is the
                     independent dataflow runtime point of comparison.

d) Same GHC RTS allocator across ALL Haskell binaries. This is the
   specific fairness fix the Euro-Par 2026 reviewer flagged: the
   original submission tuned `SUPERS_RTS_A=256m` only on the TALM
   side and left every other Haskell binary at the GHC default
   (much smaller nursery), which gave TALM-Hs an allocator advantage
   the speedup did not control for. In this revision, every GHC-
   compiled binary is invoked with `+RTS -A256m -RTS` and the TALM
   interp's internal Haskell-shim runs with `SUPERS_RTS_A=256m`.
   Concretely, the value `TALM_RTS_A=256m` (default, settable via
   env var) is threaded into ALL of:
     * sequential baseline    (seq, +RTS -A$TALM_RTS_A -RTS)
     * TALM-Hs supers         (SUPERS_RTS_A=$TALM_RTS_A on the interp)
     * GHC Strategies         (+RTS -N$P -A$TALM_RTS_A -RTS)
     * Repa                   (excluded from final master.csv, but
                               also wired identically when it ran)
     * par/pseq               (skipped in this sweep, also identical
                               when used; SKIP_PARPSEQ=1)
     * TALM-C interp          (SUPERS_RTS_A=$TALM_RTS_A — the
                               Trebuchet wrappers carry a Haskell
                               shim that still allocates)
   See scripts/lcs_wavefront/run_validated.sh, line 200 (seq), 265
   (TALM), 296 (par/pseq), 315 (STRAT). The Rust variants (TALM-Rust,
   Timely) have no GHC RTS and are not affected; the Sucuri Python
   driver also has no GHC nursery. The speedup denominator (seq) is
   the central beneficiary of this equivalence — the original
   reviewer complaint was precisely that the denominator was
   penalised by a smaller nursery than the numerator.

d2) Per-language baselines (the OTHER fairness fix). The original
    submission reported every parallel variant's speedup against the
    Haskell sequential baseline. That is incorrect for cross-language
    variants: the Ribault-C and Ribault-Rust speedups would conflate
    the dataflow-scheduler win with the GHC↔C / GHC↔Rust compute gap
    that lives outside the parallelism contribution. The Rust variants
    (Timely, Sucuri) would do the same. The honest fix is to measure
    each parallel numerator against a same-language sequential
    denominator:

      * Ribault-Hs, GHC Strat   -> seq         (Haskell, +RTS -A256m)
      * Ribault-C               -> seq_c       (C, gcc -O3 -march=native)
      * Ribault-Rust, Timely,
        Timely-Reb, Sucuri      -> seq_rust    (Rust release, raw
                                                pointers + unsafe inner
                                                loop matching the
                                                parallel leaf compute)

    All three baselines run a monolithic 2-row DP, use the identical
    LCG (seed=42, alpha=4), and produce the SAME RESULT at every N
    (cross-validation hash: 65385 / 130837 / 196253 / 261674 / 327182
    at N = 100k / 200k / 300k / 400k / 500k). Each baseline is the
    fastest same-language single-thread idiom we could write while
    still matching the parallel variant's compute backend; this gives
    the conservative (smallest) honest speedup for each tier and
    removes the cross-language inflation that the original numbers
    contained. The seq_c / seq_rust runs were collected with the same
    REPS=7, same taskset core-0 pin, and same percentile-bootstrap CI
    machinery as every other variant. See sections A2, A3 above for
    the source-level details; the per-language tables in
    tables_with_ci.txt are split into a "Haskell tier", "C tier", and
    "Rust tier" with the matching denominator quoted in each header.

e) Same thread count plumbing. GHC binaries get `+RTS -N<P> -RTS`.
   TALM (Hs and C) gets `SUPERS_RTS_N=<P>`. Timely gets `-w <P>`.
   Sucuri uses its config that spawns <P> workers in the
   free-threaded interpreter.

f) Same core pinning. Every measurement is launched via
   `taskset -c 0-(P-1) ...`. For P <= 24 this keeps work on
   physical cores. For P > 24 (we use P=32 and P=48) it extends
   into HT siblings, so the P=32 and P=48 numbers reflect SMT
   contention and are interpretively distinct from the
   pure-physical-core P=2..24 numbers — the runner's stdout
   prints an OVERSUBSCRIPTION banner at that boundary.

g) Same number of repetitions and same medianing + confidence
   intervals. REPS=7 per (variant, N, P); the point estimate
   reported in every table is the **median** of the seven runs
   (sorted ascending, position 4 of 7).

   In addition to the median, every reported speedup carries a
   **95% confidence interval (IC)** computed by **percentile
   bootstrap**: resample the 7 raw runtimes with replacement 10000
   times, take the median of each resample, then report the 2.5th
   and 97.5th percentile of that bootstrap distribution as the
   lower/upper CI bound for the per-cell variant runtime. The CI
   for a speedup (T_seq / T_variant, where T_seq is the
   language-matched baseline from section 8d2 — seq for Haskell-tier
   variants, seq_c for Ribault-C, seq_rust for Ribault-Rust /
   Timely / Timely-Reb / Sucuri) propagates uncertainty from BOTH
   numerator and denominator: in each of the 10000 bootstrap
   iterations we resample the baseline runs and the variant runs
   INDEPENDENTLY (with replacement), recompute the medians, take
   their ratio, and read the 2.5th / 97.5th percentile of the
   resulting ratio distribution.

   Independent (not paired) resampling is the correct choice here
   because the seq and variant runs were NOT collected in paired
   trials: the runner (scripts/lcs_wavefront/run_validated.sh) runs
   all 7 seq reps consecutively at the start of an N, then for each
   P runs all 7 variant reps consecutively, with no shared seed
   or shared machine-state between the i-th seq run and the i-th
   variant run. Resampling independently gives the conservative
   (slightly wider) CI that does not assume a phantom correlation;
   if seq and variant had been paired by shared run state, a
   paired resample (one index draw, used for both seq[i] and
   variant[i]) would capture that correlation and tighten the
   interval — but the experiment does not provide such pairing,
   so we do not exploit it.

   Bootstrap with 10000 resamples is overkill for N=7 (the median
   takes only one of 7 possible discrete values per resample), but
   it is the standard nonparametric choice for small samples and
   makes the CI reproducible by anyone who re-runs the analysis
   from master.csv. We chose bootstrap over a parametric Student-t
   CI because: (i) the 7-rep distribution is not symmetric — the
   Trebuchet dispatch tail and HT-cache effects produce skewed
   right tails at oversubscription; (ii) bootstrap is what most
   recent parallel-systems papers (PPoPP, ICS, ASPLOS) use for
   small REPS.

   Seven reps absorb single-rep outliers from CPU frequency jitter,
   HT contention spikes (P>24), and the Trebuchet super-dispatch
   tail. REPS=3 was used only during pre-experiment regime
   exploration / N-finding sondagem and is NOT the protocol for the
   data points reported in the paper.

h) GHC -O2 -threaded for all parallel Haskell variants; the
   sequential baseline is built without -threaded (no RTS spark
   manager overhead in the denominator). Rust crates are built
   with --release (opt-level=3, lto=thin, codegen-units=1, no
   debug info).


9. HOW HARDWARE INFLUENCED METHODOLOGY
----------------------------------------------------------------
- 24 physical / 48 logical cores: PS = {2, 4, 8, 16, 24, 32, 48}
  samples the scaling curve at five no-HT points (2, 4, 8, 16, 24)
  plus the physical-core boundary itself (P=24) and two oversubscribed
  points (32, 48). P=48 covers full HT saturation. The pure
  physical-core scaling ceiling lives at P=24; anything beyond that
  shares execution resources between two threads on the same core.

- 45 MiB shared L3 / 2 MiB per-core L2: at TARGET_BLOCK_WIDTH=1000,
  each block's working set (two scratch rows of ~1000 int64 = 16 KiB
  plus reads from boundary slices) fits in L1d. The boundary arrays
  themselves (~7-8 GiB at N=900k-1M) stream through L3; no variant
  tries to keep them L3-resident.

- 251 GiB RAM: the largest single-process working set is the Timely
  variant at N=500k holding hBound + vBound ~ 4 GiB. We have ample
  headroom for that, plus the GHC nursery (`-A256m` x 48 capabilities
  ~ 12 GiB worst-case), plus duplicate copies the Sucuri layer
  may retain. The 4 GiB swap was not used in earlier runs.

- Single NUMA node: no NUMA pinning required beyond core pinning.
  Boundary arrays are accessed by every worker; on a multi-node
  machine this would have demanded a different memory layout.

- hwp_epp governor (Intel HWP, energy-perf-bias active): CPU
  frequency is allowed to scale up to 3.9 GHz under load. We did
  not pin the governor to "performance"; this matches the user's
  default deployment environment. Variance between reps is the
  control for any frequency-scaling jitter (REPS=7 + median).

                           ribault_rust, sucuri, timely}
  seq_len   int    N
  dim_rows  int    DIM_ROWS (= P for parallel; 1 for seq)
  dim_cols  int    DIM_COLS (= ceil(N/1000), floor P; 1 for seq)
  P         int    worker count
  rep       int    1..REPS
  seconds   float  wall-clock parallel-section runtime

Speedup at (variant, N, P) =
   median(seq seconds at N over reps)
 / median(variant seconds at (N, P) over reps)

95% CI on the speedup is computed by paired percentile bootstrap
over master.csv (10000 resamples; see section 8g for the protocol).
Tables and figures in the paper carry the median speedup and the
2.5/97.5 percentile bounds as error bars / parenthetical [lo, hi].


11. FINDINGS (preliminary, descriptive)
----------------------------------------------------------------
These notes are pattern observations across the completed N values
in master.csv. They are descriptive, not paper-claim text; the
paper's discussion section will need to formalize and defend each.

11.1  TALM-Hs vs STRAT  (Haskell tier, equal compute backend)
   STRAT (parList rseq sparks) plateaus across all P at the same
   wall-clock time at each N — it never scales past a small-P
   ceiling. TALM-Hs scales monotonically with P at every N and
   beats STRAT by 1.4×-5× depending on (N, P), with the largest
   advantage at P=16-32 in the TARGET_BLOCK_WIDTH=100 regime where
   barrier overhead per anti-diagonal is non-trivial relative to
   per-block compute. The mechanism: GHC's spark queue serializes
   on the barrier between anti-diagonals; TALM's firing rule has
   no barrier and lets each worker grab the next ready block as
   soon as its two dataflow operands have arrived. This is the
   most direct evidence that the dataflow scheduler — not the
   per-block compute backend — accounts for the gap.

11.2  TALM-Rust vs Timely  (Rust tier, equal compute backend)
   In the steady-state regime (P <= 24 physical cores), TALM-Rust
   matches Timely or slightly leads (typical margins +4% to +20%).
   Both are competitive Rust-native dataflow runtimes with raw-i64
   shared-state compute. Two anomalies in Timely show up clearly
   in the data and need separate paper-level justification:

   (a) Prime-P dips. Timely's Exchange operator routes records by
       hash(coord) mod P. When P is prime (3, 7, 13, 19, 23) and
       DIM_COLS is large (multiple of 100), the hash distribution
       across workers becomes uneven — one or two workers end up
       holding a heavier share of each anti-diagonal's blocks.
       Because every block in the wavefront is on the critical
       path of its anti-diagonal (it gates the next epoch), the
       slowest worker dictates the per-epoch wall time. Concrete
       prime-P dips observed (median of 7 reps):
         * N=100k P=3 : Timely 2.34× vs P=2 9.20× and P=4 17.36×
         * N=200k P=7 : Timely 14.17× vs P=4 15.95× and P=8 29.14×
         * N=300k P=7 : Timely 18.96× vs P=4 20.69× and P=8 37.31×
         * N=300k P=19: Timely 45.10× vs P=24 81.01×
       TALM-Rust does NOT exhibit these prime dips because the
       firing rule has no Exchange and no global per-epoch barrier:
       each block fires the moment its two boundary inputs arrive,
       regardless of how the worker assignment maps to anti-diagonal
       indices. The per-block compute then dominates and scaling is
       monotonic across all primes.

   (b) Oversubscription volatility (P=32, P=48). Timely's epoch
       barrier plus the Exchange shuffle contends on HT-shared
       resources and produces high run-to-run variance — alternating
       between cache-hot spikes (e.g. P=32 N=200k speedup 60.84× vs
       TALM-Rust 42.91×) and severe dips (e.g. P=24 N=400k 32.74×
       vs TALM-Rust 88.32×). TALM-Rust's firing rule has no global
       barrier and no inter-worker shuffle, so its oversubscription
       behaviour is monotonic and predictable.

   Both (a) and (b) reduce to the same underlying mechanism: any
   global synchronisation point (epoch barrier, exchange shuffle)
   becomes a load-balancing failure mode when the workload is
   wavefront-shaped (every block on the critical path of some
   anti-diagonal). The firing rule replaces the synchronisation
   point with per-block readiness checks, which are immune to the
   distribution of work across workers — they only depend on the
   producer-to-consumer dataflow edges, which are workload-defined,
   not P-defined.

11.3  TALM-C  (C compute, no language penalty)
   TALM-C remains the absolute fastest variant across the entire
   sweep, peaking near 130× at N=400k P=16-32 (the per-block
   compute is small enough that Trebuchet's dispatch overhead
   matters but the C inner loop is still ~2× tighter than what
   rustc release emits for the same int32 DP). This represents
   the ceiling of what the Trebuchet scheduler can extract when
   the per-block compute is removed as a confound; no C-native
   dataflow runtime exists for head-to-head comparison.

11.4  Sucuri+Rust  (independent dataflow research runtime)
   Plateaus around 9-10× in the small-block regime — the Python
   orchestrator (free-threaded CPython 3.14t) dispatches blocks
   too slowly to keep workers busy when per-block compute is
   small. At larger TARGET_BLOCK_WIDTH (1000) Sucuri reached
   ~80× peaks; at TARGET=100 it ceiling-limits. Useful as an
   external dataflow framework reference, with the caveat that
   the bottleneck observed here is the Python dispatch loop, not
   the Rust leaf or the conceptual scheduler.


12. WHAT'S DELIBERATELY OMITTED
----------------------------------------------------------------
- monad-par: removed at user request after preliminary runs (N=1M
  and N=2M REPS=1) showed it as a pure-Haskell scheduler with no
  cross-language cost. The user judged the comparison structurally
  unfair to TALM-Hs (which pays Trebuchet's C-runtime cost per
  block). monad-par's role of "lower-bound Haskell-native overhead"
  is left to discussion in the paper, not as a head-to-head adversary.

- Repa: removed for the same structural reason as monad-par.
  Repa's scheduler is 100% Haskell (R.computeP uses GHC's native
  thread pool, sized by +RTS -Nx). Per-block dispatch pays no FFI
  cost; TALM-Hs pays Trebuchet's C-runtime Haskell→C→Haskell crossing
  on every block. Comparing them measures language-boundary overhead
  rather than the dataflow scheduler, the same fairness objection
  that excluded monad-par. Additionally, our Repa wavefront driver
  is non-idiomatic: it forces Repa via unsafePerformIO inside
  R.computeP, with Data.Vector.Unboxed.Mutable doing the actual
  compute — effectively using Repa as a thread pool with
  anti-diagonal barriers, not as the regular-array-combinator
  library it was designed to be. The Repa runner is kept on the
  server for sanity-checking but its rows are filtered out of
  master.csv and do not appear in the paper. STRAT is retained
  despite being similarly pure-Haskell-scheduled because
  reviewer #116C explicitly requested a spark/Strategies
  comparison ("swap the parallelism for a spark or
  strategies-based implementation").

- par/pseq (manual sparking, gen_hs_parpseq.py): plateaus at ~1.3x
  speedup for LCS wavefront regardless of P (diagonal barrier
  ceiling) and is the slowest variant to compile at large N. It is
  skipped in this sweep (SKIP_PARPSEQ=1).

- P=1 for parallel variants: the sequential baseline already
  measures the no-parallelism case. Running TALM-Hs at P=1 would
  measure the Trebuchet startup tax against itself, which is not
  what speedup is meant to capture.

- N=1M: dropped from this sweep so we can compare at fixed budget;
  earlier sondagem (N=1M REPS=3 and N=2M REPS=1) is available in
  separate result directories on the server if needed for the paper.

- Dryad / Naiad as live competitors: both are retired by Microsoft
  and not runnable on a current Linux box. Timely Dataflow stands
  in as the live representative of their paradigm.


13. KNOWN CAVEATS
----------------------------------------------------------------
- Rust variants (Sucuri leaves, Timely) inherently run faster per
  block than Haskell variants because Rust generates tighter code
  for the inner DP loop. Speedups for Sucuri/Timely against the
  Haskell sequential baseline include this language-level gap.
  We mitigate this by also reporting variant-internal scaling
  (Sucuri at P=2 vs P=48, Timely at P=2 vs P=48) which factors out
  the language constant.

- The Timely raw-pointer write pattern depends on disjoint write
  regions per block on each anti-diagonal. This invariant holds
  for the LCS block layout but would not hold for an arbitrary
  dataflow pattern; it is a workload-specific safety argument,
  not a general one.

- At the largest N here (500k) the per-block compute is ~5e6 int
  operations and the scheduler-overhead-to-compute ratio is small
  but not zero. Earlier sondagem experiments (100k -> 2M) showed
  the gap between dataflow systems and TALM-Hs narrowing
  monotonically as N grows.
